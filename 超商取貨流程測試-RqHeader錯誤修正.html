🛠️ 超商取貨流程測試 - RqHeader 錯誤修正

❌ 原始錯誤

{ "PlatformID": "", "MerchantID": "", "RpHeader": { "Timestamp": "1753115955" }, "TransCode": 703, "TransMsg": "The Object [RqHeader] cannot be null", "Data": "" }

問題分析：ECPay v2 API 要求必須包含正確的 RqHeader 參數，且不能為 null。

✅ 解決方案

改用電子地圖 API：

  • 從 v2/RedirectToLogisticsSelection 改為 Express/map
  • 使用標準表單參數而非加密的 Data 參數
  • 正確計算 CheckMacValue 檢查碼
  • 移除有問題的 RqHeader 參數

🧪 測試流程

步驟1：API 連通性測試

檢查後端服務是否正常運行

步驟2：電子地圖功能測試

開啟 ECPay 電子地圖選擇超商門市

步驟3：完整流程測試

模擬從結帳到門市選擇的完整流程

📊 技術細節

修正前（有問題的實作）：

API: https://logistics-stage.ecpay.com.tw/Express/v2/RedirectToLogisticsSelection 參數: { MerchantID: "xxx", RqHeader: "{\"Timestamp\":\"xxx\",\"Revision\":\"1.0.0\"}", Data: "加密的參數字串" } 問題: RqHeader 格式錯誤，ECPay 無法解析

修正後（正確的實作）：

API: https://logistics-stage.ecpay.com.tw/Express/map 參數: { MerchantID: "xxx", MerchantTradeNo: "MAP1234567", LogisticsType: "CVS", LogisticsSubType: "UNIMARTC2C", IsCollection: "N", ServerReplyURL: "http://localhost:9000/store/ecpay/map-callback", ExtraData: "{...}", Device: "0", CheckMacValue: "計算的檢查碼" } 優點: 標準表單參數，不需要複雜的加密流程
